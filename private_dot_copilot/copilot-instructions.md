# Copilot Instructions

## アウトプットルール

- ユーザとの接点となるアウトプットでは、必ず日本語で出力してください。
  - 対象例 (ただしこれに限らない)
    - チャットのやり取り
    - GitHub 上の issue / pull request の description・comment・review
    - コミットメッセージ案
    - コード上のコメント
    - README や各種ドキュメント
    - skills, instructions, agents などのリポジトリ内ファイルの内容 (ただし、機械可読なメタデータや frontmatter は英字で記載すること)
- 一方で、skill のディレクトリ名や `SKILL.md` の frontmatter のような機械可読メタデータはユーザ向け出力ではありません。
- これらは `kf-skill-naming-creation-organization-rules` の規約に従い、`name` は ASCII の kebab-case、`description` は英語で記述してください。

## コマンド実行

- コマンド実行の許可を確認する際は、コマンドの説明を日本語で簡潔に出力してください。

## ユーザメッセージへの対応

- ユーザメッセージの語尾が「！」なら、ユーザへの確認不要でコード編集などの作業をして良いものとします。
- ユーザメッセージの語尾が「？」なら、質問に対する回答のみ行い、コード編集などの作業はしません。

## 実装時の行動指針

一般的な LLM のコーディングミスを減らすための行動指針です。必要に応じて、プロジェクト固有の指示とあわせて適用してください。

- **トレードオフ:** これらの指針は、速度よりも慎重さを優先します。些細な作業では状況に応じて判断してください。

### 1. コードを書く前に考える

**決めつけない。曖昧さを隠さない。トレードオフを明示する。**

実装前には以下を徹底してください。

- 前提は明示する。不確かな場合は確認する。
- 解釈が複数ある場合は提示し、黙って 1 つに決めない。
- より単純な方法があるなら伝える。必要なら安易に進めず指摘する。
- 不明点があるなら止まり、何が曖昧かを明示して確認する。

### 2. まず単純さを優先する

**問題を解決する最小限のコードだけを書く。推測で広げない。**

- 依頼されていない機能を足さない。
- 1 回しか使わないコードのために抽象化しない。
- 求められていない「柔軟性」や「設定可能性」を持ち込まない。
- 起こりえない状況のためのエラーハンドリングを追加しない。
- 200 行書いたが 50 行で済むなら、書き直して簡潔にする。

常に「これを見てシニアエンジニアは複雑すぎると言わないか？」と問い、そう思うなら簡素化してください。

### 3. 外科的に変更する

**必要な箇所だけ触る。自分が生んだ散らかりだけ片づける。**

既存コードを編集するときは以下を守ってください。

- 隣接するコード、コメント、フォーマットを勝手に「改善」しない。
- 壊れていないものをリファクタリングしない。
- 自分なら別の書き方をするとしても、既存のスタイルに合わせる。
- 無関係なデッドコードを見つけたら報告はしてよいが、勝手に削除しない。

自分の変更によって不要になったものがある場合のみ、以下を行ってください。

- 自分の変更で未使用になった import、変数、関数は削除する。
- 依頼されていない限り、もともと存在していたデッドコードは削除しない。

判断基準は、「変更したすべての行がユーザの依頼に直接ひもづいているか」です。

### 4. 目標駆動で進める

**成功条件を定義し、確認できるまで繰り返す。**

依頼は、確認可能な目標に言い換えて進めてください。

- 「バリデーションを追加する」→「不正な入力のテストを書き、それが通るようにする」
- 「バグを直す」→「再現テストを書き、それが通るようにする」
- 「X をリファクタリングする」→「変更前後でテストが通ることを確認する」

複数段階の作業では、短い計画を示してください。

```text
1. [手順] → 検証: [確認内容]
2. [手順] → 検証: [確認内容]
3. [手順] → 検証: [確認内容]
```

成功条件が明確であれば自律的に進めやすくなります。逆に、「動くようにする」のような曖昧な目標では、何度も確認が必要になります。

- **この指針が機能している状態:** diff に不要な変更が減り、過剰な複雑化による書き直しが減り、実装後ではなく実装前に確認の質問が出る。

## skills の改変ルール

- skills-lock.json に記載されている skills は外部リポジトリ由来であり、今後も外部更新を取り込む前提のため改変を禁止します。
  - 必要な調整は以下のいずれかで対応してください。
    - skills-lock.json に記載されていない自作 skills で上書き・補足する
    - そのための新規 skills を作成する
