# 5文字プレフィックスの日本語コミットメッセージ

このスキルは、次のテンプレートでコミットメッセージを作るときに使います。

```text
<type_>: <subject (日本語)>

- 概要
  - <何を変えたかを記載>
- Why
  - <なぜこの対応を行なったかを記載>
```

- `type_`: 5文字固定のプレフィックス
- `subject`: 変更内容を表す日本語の簡潔な説明
- 2行目: 空行
- 3行目以降: 変更概要と理由を書く本文

## このスキルを使う場面

次のような依頼ではこのスキルを使ってください。

- コミットしてください
- コミットメッセージを考えてほしい
- 変更内容から適切な prefix を選んでほしい
- 既存のコミットメッセージをこのルールへ合わせて整形したい
- 日本語 subject のコミットメッセージ規約を適用したい
- `feat_` `fix__` `docs_` `style` `refa_` `perf_` `test_` `chore` の使い分けが必要

## 基本ルール

### 1. 出力形式は subject 行と本文で構成する

コミットメッセージは常に次の複数行形式で出力します。

```text
<type_>: <subject (日本語)>

- 概要
- Why: なぜこの対応を行なったか
```

構成ルール:

- 1行目は `type_: subject` 形式にする
- 2行目は必ず空行にする
- 3行目以降に本文を書く
- 本文は箇条書きで書く
- 少なくとも `- 概要` と `- Why` を含める

### 2. プレフィックスは5文字固定

使用できるプレフィックスは次の8種類だけです。

| Prefix  | 意味                         | 使用例                                    |
| :------ | :--------------------------- | :---------------------------------------- |
| feat\_  | 新機能の追加                 | feat\_: ユーザーログイン機能を実装        |
| fix\_\_ | バグ修正                     | fix\_\_: 検索モーダルの表示崩れを修正     |
| docs\_  | ドキュメントのみの変更       | docs\_: README.mdのセットアップ手順を更新 |
| style   | コードの意味に影響しない変更 | style: インデントをスペース4つに統一      |
| refa\_  | リファクタリング             | refa\_: Userクラスのメソッド抽出          |
| perf\_  | パフォーマンス改善           | perf\_: 画像読み込みの遅延ロード対応      |
| test\_  | テストの追加・修正           | test\_: 決済処理の単体テストを追加        |
| chore   | ビルドプロセスやツールの変更 | chore: 依存ライブラリのバージョン更新     |

プレフィックスは表記を変えず、そのまま使ってください。似た意味でも、一覧にない prefix は使わないでください。

### 3. subject は日本語で簡潔に書く

subject は変更の中心を短く説明してください。

- 日本語で書く
- 変更の主目的を一つに絞る
- 冗長な前置きは入れない
- 文末の句点は付けない
- 「修正対応」「対応を実施」のような曖昧表現より、何を変えたかを具体的に書く

良い例:

- `fix__: 検索結果が0件でもローディングが終了するよう修正`
- `refa_: 認証処理をサービス層へ分離`
- `docs_: 開発環境構築の前提条件を追記`

避ける例:

- `fix__: いろいろ修正`
- `chore: 対応`
- `feat_: 機能を追加しました`

### 4. 本文は概要と理由を明記する

3行目以降の本文には、少なくとも次の2項目を書いてください。

- `- 概要`
- `- Why: なぜこの対応を行なったか`

書き方のルール:

- 箇条書きで始める
- `概要` には何を変えたかを短く書く
- `Why` には背景、問題、意図、必要性を書く
- `Why` のラベルは `Why:` と明示する
- 本文も日本語で簡潔に書く

例:

```text
feat_: ユーザーログイン機能を実装

- 概要: メールアドレスとパスワードによるログイン機能を追加
- Why: 認証なしではマイページ機能を利用できず、利用開始までの導線が不足していたため
```

## プレフィックス選択ガイド

変更内容を見て、次の基準で最も近いものを一つ選んでください。

### feat\_

ユーザー価値のある新機能、画面、API、動作追加のときに使います。

### fix\_\_

不具合、誤動作、表示崩れ、例外、期待との差分を直したときに使います。

### docs\_

README、設計書、コメント整備など、ドキュメントだけを変更したときに使います。

### style

フォーマット、空白、lint 修正、並び順調整など、意味や挙動に影響しない変更に使います。

### refa\_

内部構造の整理、責務分割、命名改善、重複除去など、機能追加やバグ修正を含まない変更に使います。

### perf\_

速度、メモリ効率、描画性能、I/O 効率などの改善に使います。

### test\_

単体テスト、結合テスト、E2E テストの追加や修正に使います。

### chore

依存関係、CI、ビルド設定、開発ツール、スクリプトなど、ソースコード本体の振る舞いを直接変えない変更に使います。

## 判断ルール

### 1. 複数変更がある場合は主目的を優先する

1回のコミットに複数要素が含まれていても、最も重要な変更意図に合わせて prefix を1つ選んでください。

例:

- 新機能追加に伴ってテストも追加した場合は、主目的が新機能なら `feat_`
- バグ修正のためにリファクタリングした場合は、主目的が不具合修正なら `fix__`

### 2. 判断に迷うときは外から見える効果を優先する

内部的には整理でも、ユーザーから見て新しい価値や不具合修正があるなら、`feat_` や `fix__` を優先してください。

### 3. 情報が不足している場合

変更内容が曖昧で prefix を断定しにくいときは、勝手に細部を作らず、分かる範囲で最も無理のない案を出してください。ユーザーが説明も求めていない場合は、過剰な注釈は付けません。説明が必要な場面だけ、どの前提で選んだかを短く添えてください。

## 生成手順

コミットメッセージを作るときは、次の順で考えてください。

1. 変更内容から主目的を1つに要約する。
1. 8種類の prefix から最も適切なものを選ぶ。
1. 変更内容を日本語で簡潔な subject にする。
1. 本文に `概要` と `Why` を日本語で書く。
1. 1行目を `<type_>: <subject (日本語)>` の形に整える。
1. 2行目を空行にする。
1. prefix が5文字固定になっているか確認する。

## 本文テンプレート

迷ったら次の形をベースにしてください。

```text
<type_>: <subject (日本語)>

- 概要: 何を変えたか
- Why: なぜこの対応を行なったか
```

## 出力例

**例1:**
入力: ログイン失敗時にエラーメッセージが表示されない不具合を修正
出力:

```text
fix__: ログイン失敗時のエラーメッセージ表示を修正

- 概要: 認証失敗時にエラートーストが表示されるよう修正
- Why: 失敗理由が分からず再試行しづらい状態になっていたため
```

**例2:**
入力: README にローカル開発手順を追記
出力:

```text
docs_: READMEにローカル開発手順を追記

- 概要: 初回セットアップと起動確認までの手順を README に追記
- Why: 新規参加者が環境構築で詰まりやすく、参照先が分散していたため
```

**例3:**
入力: API レスポンスのキャッシュ処理を改善して一覧表示を高速化
出力:

```text
perf_: 一覧APIレスポンスのキャッシュ処理を改善

- 概要: 一覧APIのキャッシュ利用条件を見直して再取得回数を削減
- Why: 画面遷移のたびに不要な再フェッチが発生し、表示完了まで時間がかかっていたため
```

## commit message 全体サンプル

コミットメッセージ全体は次のように書きます。

```text
feat_: ユーザー招待メールの再送機能を追加

- 概要: 管理画面から招待メールを再送できる操作と API を追加
- Why: 初回送信メールの期限切れや未達時に、管理者が再招待する手段が必要だったため
```

## 最終チェック

返す前に次を確認してください。

- 1行目が `<type_>: <subject>` になっている
- 2行目が空行になっている
- 3行目以降に本文がある
- prefix が許可された8種類のどれかで、5文字固定になっている
- subject が日本語で簡潔に書かれている
- subject が変更の主目的を表している
- 本文に `- 概要` と `- Why:` が含まれている
- 本文が日本語で簡潔に書かれている
