# 文体・チェック・例

## subject の書き方

- 日本語で書く。
- 変更の主目的を 1 つに絞る。
- 文末の句点は付けない。
- 体言止めを優先する。
- 敬語は使わない。
- 「対応」「いろいろ修正」のような曖昧表現を避ける。

良い例:

- `fix__: 検索結果0件時のローディング終了不備を修正`
- `refa_: 認証処理のサービス層分離`
- `docs_: 開発環境構築の前提条件を追記`

避ける例:

- `fix__: いろいろ修正`
- `chore: 対応`
- `feat_: 機能を追加しました`

## 本文の書き方

- 長文より短い箇条書きを優先する。
- 1 つの箇条書きには 1 論点だけを書く。
- 接続詞や前置きは最小限にする。
- `概要` には何を変えたかを書く。
- `Why` には背景、問題、意図、必要性を書く。

## 出力例

```text
docs_: READMEにローカル開発手順を追記

- 概要:
  - 初回セットアップ手順を追記
  - 起動確認手順を追記
- Why:
  - 新規参加者が環境構築で詰まりやすい状況
  - 参照先分散の解消
```

```text
perf_: 一覧APIレスポンスのキャッシュ処理を改善

- 概要:
  - 一覧APIのキャッシュ利用条件を見直し
  - 不要な再取得回数を削減
- Why:
  - 画面遷移ごとの再フェッチ発生
  - 表示完了までの待ち時間短縮
```

## 最終チェック

- 1 行目が `<type_>: <subject>` になっている。
- 2 行目が空行になっている。
- 3 行目以降に本文がある。
- prefix が許可された 8 種類のどれかで、5 文字固定になっている。
- subject が日本語で簡潔に書かれている。
- subject が変更の主目的を表している。
- 本文に `- 概要` と `- Why:` が含まれている。
- 全体が端的で、長文は箇条書きへ分割されている。
