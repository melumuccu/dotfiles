# prefix 選択ガイド

## 使える prefix

| Prefix | 意味 | 主な用途 |
| :-- | :-- | :-- |
| feat_ | 新機能の追加 | 新しい操作、画面、API、対応環境の追加 |
| fix__ | バグ修正 | 誤動作、表示崩れ、例外、回帰の修正 |
| docs_ | ドキュメント変更 | README、設計書、コメント整備 |
| style | 非機能の整形 | フォーマット、空白、並び替え |
| refa_ | リファクタリング | 責務分割、重複除去、命名改善 |
| perf_ | 性能改善 | 速度、メモリ、I/O 効率の改善 |
| test_ | テスト追加・修正 | 単体、結合、E2E テスト |
| chore | ツール・運用変更 | 依存更新、CI、ビルド、開発運用 |

## 選び方

1. 変更の主目的を 1 つに要約する。
1. その主目的に最も近い prefix を 1 つ選ぶ。
1. 複数要素がある場合も、主目的を優先する。

## 判断ルール

### feat_

- これまで無かった操作導線、設定、対応プラットフォームを増やすときに使う。
- PC にはあるがスマホには無かった導線を追加するような拡張も `feat_` に寄せる。

### fix__

- 既に提供しているはずの挙動が壊れている、仕様とずれている、回帰した場合に使う。
- 未対応環境への新規対応や新導線の追加は、問題解消の文脈があっても原則 `feat_` を優先する。

### docs_

- 変更がドキュメントだけで閉じるときに使う。

### style

- コードの意味や挙動に影響しない整形だけのときに使う。

### refa_

- 機能追加やバグ修正を主目的にせず、内部構造だけを整理するときに使う。

### perf_

- ユーザー体験に直結する性能改善を主目的にするときに使う。

### test_

- テスト追加やテスト修正が主目的のときに使う。

### chore

- ソース本体の振る舞いを直接変えない、依存・ツール・運用変更に使う。

## 迷ったとき

- 内部整理でも、外から見える効果が新機能なら `feat_` を優先する。
- 内部整理でも、外から見える効果が不具合修正なら `fix__` を優先する。
- 情報が不足して断定しにくいときは、勝手に細部を作らず、分かる範囲で最も無理のないものを選ぶ。
