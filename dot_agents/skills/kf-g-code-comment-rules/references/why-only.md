# WHY のみ

## 基本方針

- コード内コメントは常に WHY だけ書く。WHAT はコードから読み取れる前提にする
- コメントは、コードだけでは読めない文脈を未来の作業者に渡すためのもの
- 処理内容の説明が必要に見えるなら、まず命名、関数分割、型、テストで読みやすくできないかを優先する

## コメントしてよい内容

次のような、コードから読み取れない理由や背景だけをコメントに残す。

- なぜそのライブラリ、API、依存関係を選んだか
- なぜ単純に見える別実装を避けたか
- なぜこの順序、条件分岐、待機、再試行、フォールバックが必要か
- なぜ互換性、性能、セキュリティ、運用上の制約を優先しているか
- なぜ外部サービス、ブラウザ、Docker、ファイル形式などの癖に合わせているか
- なぜ一見不要に見える処理を残しているか
- 過去の不具合や回帰を避けるための判断理由

## コメントしない内容

次のような WHAT の説明は書かない。既にある場合は、周辺変更の範囲で削除または WHY に置き換える。

- 関数名、変数名、型、制御構造を読めば分かる処理説明
- `if`、`map`、`filter`、`return` などの逐語的な説明
- 「画像を変換する」「レスポンスを返す」などコードと同じ粒度の説明
- 実装と同期し続ける必要があるだけのコメント
- TODO の形をした曖昧な願望や、担当者の意図が分からないメモ

## 判断手順

コメントを追加または維持する前に、次を確認する。

1. そのコメントは WHY を説明しているか
1. コメントなしだと、将来の作業者が誤って削除、単純化、置換しそうか
1. コードの命名や構造を改善するほうが自然ではないか

1 つ目が満たせない場合、そのコメントは書かない。3 つ目が自然な場合、コメントよりコード改善を優先する。

## 例

避ける:

```ts
// 画像を JPEG に変換する
const converted = await convertToJpeg(image);
```

よい:

```ts
// HEIC はブラウザ表示が不安定なため、JPEG に寄せる。
const converted = await convertToJpeg(image);
```

避ける:

```ts
// アセット ID がなければ 400 を返す
if (!assetId) {
  return error(400, "assetId is required");
}
```

よい:

```ts
// Immich API への空 ID 転送は 500 として返るため、利用者向けにはここで入力エラーに正規化する。
if (!assetId) {
  return error(400, "assetId is required");
}
```

## レビュー時の扱い

- WHAT コメントを見つけたら削除候補として扱う
- WHY が必要なのにコメントがない場合は、コメント追加だけでなく、より明確な名前や構造にできないかも確認する
